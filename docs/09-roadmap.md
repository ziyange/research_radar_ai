# 09 路线图

版本：v0.1  
日期：2026-06-14  
状态：MVP 基线

## 1. 阶段原则

产品路线按用户价值闭环推进：

1. 先验证推荐准确。
2. 再增强全文研读。
3. 再构建科研知识系统。
4. 再做本地连接器和机构能力。
5. 最后扩展增长、商业化和私有部署。

## 2. Phase 0：文档基线

目标：

- 建立需求、架构、接口、数据模型、测试和风险文档。
- 锁定 MVP 验收目标。

交付：

- `docs/` 文档体系。
- 稳定需求编号。
- MVP 技术栈。
- 验收矩阵。
- 后续能力追踪文档，承接未进入 MVP 的 `RR-FUTURE-*`。

退出标准：

- 产品、研发、测试都能根据文档理解 MVP 范围。
- 后续开发可以按需求编号拆任务。

## 3. Phase 1：MVP 闭环

目标：

- 验证研究画像、自动检索、推荐、反馈和推送闭环。

范围：

- 一句话冷启动。
- 基石论文上传。
- 研究画像生成和确认。
- OpenAlex、Crossref、Semantic Scholar、arXiv 中至少 2 个数据源。
- 跨源排重。
- 个性化推荐。
- 快速 AI 分析。
- 知识库基础状态。
- 每日雷达和每周周报。
- 成本记录和额度。

退出标准：

- 5 至 10 个种子研究方向跑通。
- 推荐结果获得种子用户正向反馈。
- 用户反馈能影响后续推荐。
- 成本可控。

## 4. Phase 2：全文研读

目标：

- 从摘要级推荐升级到全文证据分析。

范围：

- PDF 上传和文本解析。
- 章节识别。
- 表格提取。
- 图表标题和说明提取。
- 证据定位。
- 实验参数提取。
- 全文问答。
- 多论文对比。
- Zotero 导入导出。
- 可评估 `RR-FUTURE-002` 月报基础版，但不得阻塞全文研读主线。

退出标准：

- 用户能围绕单篇论文获得可追溯全文研读报告。
- 实验参数提取对首批学科方向有可用准确率。
- 全文分析成本可控。

## 5. Phase 3：科研知识系统

目标：

- 将文献结果沉淀为结构化科研知识。

范围：

- 材料库。
- 方法库。
- 参数库。
- 性能库。
- 主张与证据库。
- 冲突结论。
- 研究空白。
- 技术路线图。
- 高级科研对话。
- 承接 `RR-FUTURE-003` 中文学术数据源、`RR-FUTURE-004` 用户自定义来源、`RR-FUTURE-005` 学科专业数据库的数据源扩展评估。

退出标准：

- 用户可以从知识库回答“哪些文献支持某观点”。
- 用户可以比较关键参数和性能。
- 系统能生成可追溯研究空白候选。

## 6. Phase 4：本地连接器

目标：

- 连接用户真实阅读和机构权限环境。

范围：

- 浏览器扩展。
- 当前论文页面识别。
- DOI 和元数据提取。
- 用户本地合法获取 PDF 导入。
- Zotero 同步。
- 桌面端。
- 文件夹监控。
- 本地隐私模式。
- 承接 `RR-FUTURE-006` 移动端能力和 `RR-FUTURE-009` 高级数据源合规策略。

退出标准：

- 用户能从浏览器一键保存论文到项目。
- 用户不需要向云端提交学校账号密码。
- 本地 PDF 可以自动进入解析队列。

## 7. Phase 5：增长与商业化

目标：

- 形成产品传播和付费转化。

范围：

- 公开研读报告。
- 研究方向周报。
- 文献发展时间线。
- 组会文献包。
- 邀请机制。
- 研点奖励。
- 会员体系。
- 付费转化。
- 承接 `RR-FUTURE-001` 多渠道推送和 `RR-FUTURE-007` 高级增长资产。

退出标准：

- 分享内容能带来有效注册。
- 用户愿意为标准研读、全文深读或高级推荐付费。
- 研点体系不激励低质量行为。

## 8. Phase 6：课题组与机构版

目标：

- 从个人工具升级为团队和机构科研知识基础设施。

范围：

- 课题组空间。
- 导师后台。
- 阅读任务。
- 共享知识库。
- 成员权限。
- 团队额度。
- 高校机构版。
- 企业研发版。
- 私有部署。
- 承接 `RR-FUTURE-008` 企业与机构集成。

退出标准：

- 课题组可协作沉淀文献和研究证据。
- 导师可查看学生阅读和知识沉淀质量。
- 企业或机构具备数据隔离、审计和部署方案。

## 9. 暂缓事项

以下事项只有在明确商业或用户价值后再进入规划：

- 大规模多智能体自治。产品成熟后只允许按 `RR-FUTURE-010` 引入受控 Agent，由工作流调度，不允许无限自主循环。
- 自动写完整论文。
- 不受控的自动全文下载。
- 非合规商业数据库采集。
- 全学科一次性覆盖。

长期能力统一维护在 [11-future-capability-backlog.md](./11-future-capability-backlog.md)。这些能力来自原始产品方案，但未进入 MVP；进入开发前必须升级为具体阶段需求编号和验收测试项。
